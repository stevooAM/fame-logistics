**UserTable.tsx** — AG Grid data table:
- Use AG Grid Community (already installed) for the user list
- Columns: Username, Full Name (first + last), Email, Role (badge), Status (Active/Inactive badge), Last Login (formatted date), Actions (edit + deactivate/activate buttons)

- Search bar above grid: debounced input that passes ?search= to API
- Active/Inactive filter toggle

- Uses shadcn Dialog component
- React Hook Form + Zod validation

- Zod schema: username required (3-30 chars, alphanumeric+underscore), email required (valid email), first_name required, last_name required, role_id required

**TempPasswordDialog.tsx** — One-time temp password display:
- Modal showing the auto-generated temp password after user creation
- Clear instruction: "This password will only be shown once. Please copy and share it with the user."
- Copy-to-clipboard button
- "Done" button to close

Implement the forced password change flow per locked decision: users with is_force_password_change=true must change their password before accessing any dashboard page.

**frontend/src/app/(dashboard)/admin/change-password/page.tsx:**
- "use client" directive
- Page title: "Change Your Password"
- Subtitle: "You must change your temporary password before continuing."
- React Hook Form + Zod validation with fields:
  - Current Password (required, password input)
  - New Password (required, min 8 chars, must contain uppercase + lowercase + digit)
  - Confirm Password (required, must match new password)
- On submit: POST /api/users/change-password/ with {current_password, new_password, confirm_password} using apiFetch
- On success: update the AuthContext user data to set is_force_password_change=false, then redirect to /dashboard using router.push
- On error: show error message from API (e.g., "Current password is incorrect")
- Minimal layout — no sidebar navigation visible (user should not navigate away)
- Design: centered card layout, max-w-md, clean and focused

**frontend/src/contexts/AuthContext.tsx** — Add force password change redirect guard:
- In the existing AuthContext/AuthProvider, check if the authenticated user has is_force_password_change === true
- The user object from the auth response (login or /api/auth/me/) should include is_force_password_change field. If the backend login response includes this field in the user profile data, use it directly.
- Add logic: after authentication is confirmed and user data is loaded, if user.is_force_password_change is true AND the current path is NOT /admin/change-password, redirect to /admin/change-password
- Use Next.js useRouter and usePathname for redirect detection
- Add a method to update user state locally (e.g., setUser or updateUser) so the change-password page can clear the flag in-memory after successful password change without a full page reload
- Do NOT redirect if already on the change-password page (avoid infinite redirect loop)
