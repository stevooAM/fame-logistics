  <context>

  <options>
    <option id="railway">
      <name>Railway</name>
      <pros>Fastest setup (Dockerfile auto-detected via railway.toml); managed Postgres + Redis plugins; predictable pricing; simple CLI; Railway has a good developer experience and is widely used for Django.</pros>
      <cons>Paid from $5/mo minimum; no free tier anymore; based in US (adds ~100-200ms for Ghana users — acceptable).</cons>
    </option>
    <option id="render">
      <name>Render</name>
      <pros>Blueprint-based deploys (render.yaml); free tier available for static sites; Docker support; managed Postgres + Redis; US and EU regions.</pros>
      <cons>Free web service sleeps after 15min inactivity (not acceptable for prod — use paid $7/mo starter); deploy times are slower than Railway.</cons>
    </option>
    <option id="vps">
      <name>VPS (Ubuntu + Nginx)</name>
      <pros>Maximum control; cheapest at scale ($5-10/mo for DigitalOcean/Hetzner droplet); can put the server in a region close to Ghana (Hetzner has EU, DigitalOcean has Frankfurt).</pros>
      <cons>Self-managed OS patches, backups, uptime monitoring; Nginx + certbot + Docker all need to stay configured; higher operational load.</cons>
    </option>

<task type="auto">
  <name>Task 2: Execute deployment automation for the chosen target</name>
  <files>.planning/phases/10-security-hardening-launch/10-06-DEPLOY-LOG.md</files>
  <action>
  Based on the choice in Task 1, run the automation steps below. STOP and ask the user whenever an external dashboard action is unavoidable (account creation, DATABASE_URL copy-paste, etc.) — do not fabricate credentials.

  **If `railway`:**
  1. Ensure Railway CLI: `which railway || npm i -g @railway/cli`
  2. `railway whoami` — if not logged in, tell the user to run `railway login` in their terminal and pause.
  3. From repo root: `railway link` (or `railway init` if no project exists) — interactive; user confirms project.
  4. Check that Postgres and Redis services exist on the project: `railway status`. If missing, stop and tell the user to add them via the Railway dashboard (cannot be scripted reliably).
  5. Set env vars (paste the variable names, ask user for values once):
     ```
     railway variables set DJANGO_DEBUG=False
     railway variables set DJANGO_SECRET_KEY=<user provides>
     railway variables set DJANGO_ALLOWED_HOSTS=<user provides>
     railway variables set CSRF_TRUSTED_ORIGINS=<user provides>
     railway variables set CORS_ALLOWED_ORIGINS=<user provides>
     railway variables set DJANGO_ADMIN_ENABLED=False
     ```
  6. Trigger a deploy: `railway up`
  7. Fetch the public URL: `railway domain` — record it in DEPLOY-LOG.md.
  8. Seed 197 customers: `railway run python manage.py seed_customers`
  9. Probe `/api/health/`: `curl -fsSL <railway-url>/api/health/` — expect 200. Record the response.

  **If `render`:**
  Render does not have a deploy CLI on par with Railway; most actions happen in the dashboard.
  1. Verify repo is on GitHub: `git remote -v | grep github.com`. If not, stop and ask the user to push.
  2. Tell the user: "Go to https://dashboard.render.com/select-repo → Blueprint → select this repo → Render will read `deploy/render.yaml` and provision everything."
  3. Tell the user which env vars need manual values (DJANGO_ALLOWED_HOSTS, CSRF_TRUSTED_ORIGINS, CORS_ALLOWED_ORIGINS) — these are `sync: false` so Render asks for them during blueprint setup.
  4. Wait for user to paste back the backend URL, then: `curl -fsSL <render-url>/api/health/` to verify.
  5. Ask the user to open the Render Shell in the web service and run `python manage.py seed_customers`.

  **If `vps`:**
  1. Ask the user for SSH access details OR ask them to run the steps from `deploy/README.md` Option C on the box. Claude's tools do not SSH for safety — walk the user through it interactively.
  2. After they report success, curl the URL to verify.

  **Always (all three):**
  Deploy frontend to Vercel.
  1. `cd frontend && which vercel || npm i -g vercel`
  2. `vercel whoami` — if not authenticated, ask user to run `vercel login` and pause.
  3. `vercel link` (inside `frontend/`).
  4. Set `NEXT_PUBLIC_API_URL` on Vercel:
     ```
     vercel env add NEXT_PUBLIC_API_URL production
     ```
     (Claude runs the command; user pastes the backend URL + `/api` when prompted.)
  5. `vercel --prod --yes`
  6. Record the Vercel URL.
