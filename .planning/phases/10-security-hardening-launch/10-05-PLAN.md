  <action>
  Many PaaS providers (Railway, Render, Heroku, Fly) inject a single `DATABASE_URL`. Add support for it in `backend/config/settings.py` WITHOUT introducing a new dependency.

  Locate the `DATABASES = {...}` block and REPLACE it with:

  ```python
  # Database
  # Prefer DATABASE_URL (set by Railway/Render/Heroku) if present; fall back to discrete POSTGRES_* vars for local Docker.
  import urllib.parse as _urlparse

  _db_url = os.environ.get("DATABASE_URL", "").strip()
  if _db_url:
      _p = _urlparse.urlparse(_db_url)
      DATABASES = {
          "default": {
              "ENGINE": "django.db.backends.postgresql",
              "NAME": (_p.path or "/").lstrip("/"),
              "USER": _urlparse.unquote(_p.username or ""),
              "PASSWORD": _urlparse.unquote(_p.password or ""),
              "HOST": _p.hostname or "",
              "PORT": str(_p.port or 5432),
              "CONN_MAX_AGE": 60,
              "OPTIONS": {"sslmode": os.environ.get("DATABASE_SSLMODE", "require")},
          }
      }
  else:
      DATABASES = {
          "default": {
              "ENGINE": "django.db.backends.postgresql",
              "NAME": os.environ.get("POSTGRES_DB", "fame_fms"),
              "USER": os.environ.get("POSTGRES_USER", "fame_user"),
              "PASSWORD": os.environ.get("POSTGRES_PASSWORD", "fame_dev_password"),
              "HOST": os.environ.get("POSTGRES_HOST", "db"),
              "PORT": os.environ.get("POSTGRES_PORT", "5432"),
          }
      }
  ```

  Also add these two start-time toggles right after the DATABASES block (so Procfile/railway start commands stay short):
  ```python
  DJANGO_RUN_MIGRATIONS_ON_START = os.environ.get("DJANGO_RUN_MIGRATIONS_ON_START", "False") == "True"
  DJANGO_RUN_COLLECTSTATIC_ON_START = os.environ.get("DJANGO_RUN_COLLECTSTATIC_ON_START", "False") == "True"
  ```

  Do NOT add new items to `requirements.txt`. `urllib.parse` is stdlib. gunicorn is already in requirements.txt.

  **deploy/railway.toml**
  ```toml
  # Railway configuration for the Django backend.
  # Docs: https://docs.railway.app/reference/config-as-code
  [build]
  builder = "DOCKERFILE"
  dockerfilePath = "backend/Dockerfile"

  [deploy]
  # Migrate + collectstatic + boot gunicorn in a single start line.
  # gunicorn binds to Railway's $PORT. 3 workers is a sane default for <50 users.
  startCommand = "python manage.py migrate --no-input && python manage.py collectstatic --no-input && gunicorn config.wsgi:application --bind 0.0.0.0:$PORT --workers 3 --timeout 60 --access-logfile -"
  healthcheckPath = "/api/health/"
  healthcheckTimeout = 30
  restartPolicyType = "ON_FAILURE"
  restartPolicyMaxRetries = 3

  # NOTE:
  # Redis and Postgres must be provisioned as separate Railway services via the
  # Railway dashboard (Add Service → Database → Postgres / Redis). Their
  # connection strings are auto-injected as DATABASE_URL and REDIS_URL.
  ```

  **deploy/render.yaml**
  ```yaml
  # Render blueprint for the Django backend + managed Postgres + Redis.
  # Docs: https://render.com/docs/blueprint-spec
  services:
    - type: web
      name: fame-fms-backend
      runtime: docker
      dockerfilePath: ./backend/Dockerfile
      dockerContext: ./backend
      region: oregon
      plan: starter
      healthCheckPath: /api/health/
      envVars:
        - key: DJANGO_DEBUG
          value: "False"
        - key: DJANGO_SECRET_KEY
          generateValue: true
        - key: DJANGO_ALLOWED_HOSTS
          sync: false
        - key: CSRF_TRUSTED_ORIGINS
          sync: false
        - key: CORS_ALLOWED_ORIGINS
          sync: false
        - key: DJANGO_ADMIN_ENABLED
          value: "False"
        - key: DATABASE_URL
          fromDatabase:
            name: fame-fms-postgres
            property: connectionString
        - key: REDIS_URL
          fromService:
            type: redis
            name: fame-fms-redis
            property: connectionString
      dockerCommand: >-
        sh -c "python manage.py migrate --no-input
        && python manage.py collectstatic --no-input
        && gunicorn config.wsgi:application --bind 0.0.0.0:$PORT --workers 3 --timeout 60 --access-logfile -"

    - type: redis
      name: fame-fms-redis
      region: oregon
      plan: starter
      ipAllowList: []

  databases:
    - name: fame-fms-postgres
      region: oregon
      plan: starter
      postgresMajorVersion: 16
  ```

  **deploy/Procfile**
  ```
  web: python manage.py migrate --no-input && python manage.py collectstatic --no-input && gunicorn config.wsgi:application --bind 0.0.0.0:$PORT --workers 3 --timeout 60 --access-logfile -
  worker: celery -A config worker --loglevel=info
  ```

  **deploy/nginx.example.conf** (for the VPS option only — ignore if using Railway or Render)
  ```nginx
  # /etc/nginx/sites-available/fame-fms.conf
  # 1) HTTP → HTTPS redirect
  server {
      listen 80;
      listen [::]:80;
      server_name api.famelogistics.com;
      return 301 https://$host$request_uri;
  }

  # 2) HTTPS reverse proxy to gunicorn
  server {
      listen 443 ssl http2;
      listen [::]:443 ssl http2;
      server_name api.famelogistics.com;

      # Managed by certbot (Let's Encrypt)
      ssl_certificate     /etc/letsencrypt/live/api.famelogistics.com/fullchain.pem;
      ssl_certificate_key /etc/letsencrypt/live/api.famelogistics.com/privkey.pem;
      ssl_protocols       TLSv1.2 TLSv1.3;
      ssl_prefer_server_ciphers on;

      # Baseline security headers (Django also sets these; belt + braces)
      add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
      add_header X-Content-Type-Options "nosniff" always;
      add_header X-Frame-Options "DENY" always;
      add_header Referrer-Policy "strict-origin-when-cross-origin" always;

      client_max_body_size 20M;

      location /static/ {
          alias /opt/fame-fms/backend/staticfiles/;
          expires 7d;
          access_log off;
      }

      location / {
          proxy_pass http://127.0.0.1:8000;
          proxy_set_header Host $host;
          proxy_set_header X-Real-IP $remote_addr;
          proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
          proxy_set_header X-Forwarded-Proto $scheme;
          proxy_read_timeout 60s;
      }
  }
  ```

  Notes:
  - The Nginx `add_header ... always` directive ensures headers ship on 4xx/5xx responses too.
  - The Vercel headers mirror next.config.js (defense in depth — Vercel's edge layer emits them even if a route misses the Next header hook).
  - `client_max_body_size 20M` in Nginx accommodates document uploads.

  ```markdown
  # Deploying Fame FMS

  This directory has ready-to-use configs for three production hosting options.
  Pick ONE for the backend; the frontend always goes on Vercel.

  | Option | Backend config | Postgres | Redis | Complexity |
  |--------|----------------|----------|-------|------------|
  | A. Railway (recommended default) | `railway.toml` | Railway plugin | Railway plugin | Low |
  | B. Render | `render.yaml` | Render database | Render redis | Low |
  | C. VPS (Ubuntu + Nginx) | `nginx.example.conf` + `Procfile` | Self-managed | Self-managed | High |

  ---

  ## Required env vars (applies to all options)

  | Var | Value example | Notes |
  |-----|---------------|-------|
  | `DJANGO_DEBUG` | `False` | MUST be False in prod |
  | `DJANGO_SECRET_KEY` | 50+ random chars | `python -c "import secrets; print(secrets.token_urlsafe(64))"` |
  | `DJANGO_ALLOWED_HOSTS` | `api.famelogistics.com` | Comma-separated |
  | `DJANGO_ADMIN_ENABLED` | `False` | Keep admin off in prod |
  | `CSRF_TRUSTED_ORIGINS` | `https://app.famelogistics.com` | Frontend origin(s) |
  | `CORS_ALLOWED_ORIGINS` | `https://app.famelogistics.com` | Frontend origin(s) |
  | `DATABASE_URL` | injected by platform | Railway/Render set this automatically |
  | `REDIS_URL` | injected by platform | Railway/Render set this automatically |
  | `DJANGO_RUN_MIGRATIONS_ON_START` | `True` | Only if you don't run `migrate` in the start command |
  | `AWS_*` (optional) | cloud storage creds | Only needed once file uploads ship |

  **Frontend (Vercel) env vars:**
  - `NEXT_PUBLIC_API_URL` = `https://api.famelogistics.com/api`

  ---

  ## Option A: Railway

  1. `npm i -g @railway/cli && railway login`
  2. From repo root: `railway init` (create a new project, link it to this repo).
  3. In the Railway dashboard for this project: **+ New → Database → Postgres**, then **+ New → Database → Redis**. Railway will inject `DATABASE_URL` and `REDIS_URL`.
  4. Set the required env vars in Railway **Variables** (see table above).
  5. Railway auto-detects `deploy/railway.toml` and deploys the backend using `backend/Dockerfile`.
  6. Once deployed, grab the public URL and attach your custom domain via **Settings → Domains**.
  7. Seed 197 customers: `railway run python manage.py seed_customers` (or run once from the Railway Run shell).

  ## Option B: Render

  1. Push the repo to GitHub.
  2. In Render: **New → Blueprint** → select the repo → Render reads `deploy/render.yaml` automatically.
  3. Render provisions the web service, Postgres, and Redis in one click.
  4. Fill in the `sync: false` env vars (`DJANGO_ALLOWED_HOSTS`, `CSRF_TRUSTED_ORIGINS`, `CORS_ALLOWED_ORIGINS`).
  5. Attach custom domain in the web service settings.
  6. Seed 197 customers via the Render Shell: `python manage.py seed_customers`.

  ## Option C: VPS (Ubuntu + Nginx + Let's Encrypt)

  1. Provision an Ubuntu 22.04 server, install Docker + docker-compose.
  2. `git clone` the repo into `/opt/fame-fms`.
  3. Copy `.env.example` → `.env` and fill in production values (use real secret keys and domain names).
  4. Bring the stack up: `docker compose up -d --build`.
  5. Install Nginx: `apt install nginx certbot python3-certbot-nginx`.
  6. Copy `deploy/nginx.example.conf` → `/etc/nginx/sites-available/fame-fms.conf`; edit server_name; `ln -s` to `sites-enabled/`.
  7. Issue Let's Encrypt: `certbot --nginx -d api.famelogistics.com`.
  8. `systemctl reload nginx`.
  9. Seed customers: `docker compose exec backend python manage.py seed_customers`.

  ---

  ## Frontend (Vercel) — same for all backend options

  1. `npm i -g vercel && vercel login`
  2. From `frontend/`: `vercel link` (or `vercel` to deploy interactively the first time).
  3. Set `NEXT_PUBLIC_API_URL` in Vercel project env (all environments).
  4. `vercel --prod` to ship.
  5. Attach custom domain via Vercel → Project → Domains.

  ---

  ## Post-deploy verification (runs in plan 10-07)

  - [ ] securityheaders.com scan → A grade
  - [ ] Lighthouse on dashboard over throttled 3G → <2s
  - [ ] End-to-end smoke test: login → create customer → create job → approve → invoice → report
  ```
