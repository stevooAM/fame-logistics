<task type="checkpoint:human-action" gate="blocking">
  <name>Task 1: Cut over DNS for the production domains</name>
  <action>Point app.famelogistics.com (frontend) and api.famelogistics.com (backend) at the production services.</action>
  <instructions>
  1. **Frontend (Vercel)**
     - In the Vercel dashboard → Project → Settings → Domains → Add `app.famelogistics.com`.
     - Vercel shows you the exact CNAME record to create.
     - In your registrar's DNS panel, add the CNAME. Save.
     - Vercel auto-issues a Let's Encrypt cert once DNS resolves (usually 1-5 min).

  2. **Backend (Railway / Render / VPS)** — pick the one from 10-06:
     - **Railway:** Project → Settings → Domains → Custom Domain → add `api.famelogistics.com`. Railway shows the CNAME target; add it at your registrar.
     - **Render:** Web Service → Settings → Custom Domains → add `api.famelogistics.com`. Follow Render's instructions for the CNAME/A record.
     - **VPS:** Add an A record `api.famelogistics.com` → your VPS's public IP. Then on the box run `certbot --nginx -d api.famelogistics.com` to issue the cert (the Nginx config from 10-05 is already in place).

  3. Wait for DNS to propagate. Check with `dig app.famelogistics.com +short` and `dig api.famelogistics.com +short`, or use https://dnschecker.org.

  4. Once both domains resolve with a valid cert:
     - `curl -fsSLI https://app.famelogistics.com/login | head -1` → HTTP 200
     - `curl -fsSLI https://api.famelogistics.com/api/health/ | head -1` → HTTP 200

  5. **Update prod env vars to the custom domains**:
     - Backend host: set `DJANGO_ALLOWED_HOSTS=api.famelogistics.com`, `CSRF_TRUSTED_ORIGINS=https://app.famelogistics.com`, `CORS_ALLOWED_ORIGINS=https://app.famelogistics.com`. Redeploy.
     - Vercel: set `NEXT_PUBLIC_API_URL=https://api.famelogistics.com/api`. Redeploy (`vercel --prod --yes` from `frontend/`).

  Report back with both custom-domain URLs verified over HTTPS.
  </instructions>
  <resume-signal>Paste the two `curl -I` outputs (or equivalent) proving both custom domains respond 200 over HTTPS.</resume-signal>
</task>

<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: Run SSL Labs, securityheaders.com, and Lighthouse</name>
  <what-built>
  Custom domains are live with TLS. Now verify the external graders give A-grade scores and Lighthouse meets the 3G performance budgets.
  </what-built>
  <how-to-verify>
  1. **SSL Labs** — run on BOTH domains:
     - https://www.ssllabs.com/ssltest/analyze.html?d=app.famelogistics.com
     - https://www.ssllabs.com/ssltest/analyze.html?d=api.famelogistics.com
     - Required: grade A or A+. If B or lower, note the reason (usually: missing HSTS, weak ciphers, cert chain issue). Fix and re-run.

  2. **securityheaders.com** — run on BOTH domains:
     - https://securityheaders.com/?q=https://app.famelogistics.com
     - https://securityheaders.com/?q=https://api.famelogistics.com
     - Required: grade A or A+. Score is based on the headers set in plans 10-01 and 10-02.

  3. **Lighthouse performance (3G)** — in Chrome DevTools:
     - Open https://app.famelogistics.com, log in, land on the dashboard.
     - DevTools → Lighthouse → Mobile → Performance → check "Simulated throttling (Slow 4G)" or use DevTools Performance tab Network → "Slow 3G" + "CPU 4x slowdown".
     - Target: Dashboard LCP < 2.0s, customer list initial render < 1.0s, filter response < 500ms.
     - Record the numbers. If a metric misses, note whether it's blocked by backend latency (likely — re-test with backend cache warm) or frontend JS (bundle analysis needed).

  4. **HTTP → HTTPS redirect** — on each domain:
     - `curl -I http://app.famelogistics.com/` → 301 or 308 to https://
     - `curl -I http://api.famelogistics.com/api/health/` → 301 or 308 to https://

  Write the scores into `.planning/phases/10-security-hardening-launch/10-07-LAUNCH-CHECKLIST.md` (create the file if it doesn't exist yet; see Task 3 structure).
