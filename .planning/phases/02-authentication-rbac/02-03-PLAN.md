    1. Add to `backend/core/models.py`:
       - `PasswordResetToken` model: user FK, token (UUID, unique, indexed), expires_at (DateTimeField), is_used (BooleanField default=False)
       - Add migration `0003_password_reset_token.py` (manual)

    2. Add to `backend/core/serializers.py` (append to existing file from 02-01):
       - `LoginSerializer`: username, password, remember_me (BooleanField, default=False)
       - `PasswordResetRequestSerializer`: email (EmailField)
       - `PasswordResetConfirmSerializer`: token (UUIDField), new_password (CharField, validated against AUTH_PASSWORD_VALIDATORS)
       - `PasswordChangeSerializer`: old_password, new_password (validated against validators)
       - `UserProfileSerializer`: id, username, email, first_name, last_name, role (nested: id, name), is_force_password_change

    3. Update `backend/core/views.py` (keep existing health_check, add auth views):
       - `LoginView` (AllowAny):
         - POST: Authenticate with username/password via `django.contrib.auth.authenticate()`
         - On success: generate JWT tokens using `CustomTokenObtainPairSerializer.get_token(user)`
         - Set access_token as HttpOnly cookie (max_age = 900 for 15min, or 604800 for 7 days if remember_me=True)
         - Set refresh_token as HttpOnly cookie (max_age = 604800 for 7 days always)
         - Set cookie flags: HttpOnly=True, Secure=(not DEBUG), SameSite="Lax", Path="/"
         - Return JSON: {user: UserProfileSerializer data}
         - On failure: return 401 with {"error": "Invalid credentials"} — generic message per AUTH-05 (don't reveal which field is wrong)
         - Log LOGIN action to AuditLog with IP address

       - `LogoutView` (IsAuthenticated):
         - POST: Read refresh_token from cookie, blacklist it via OutstandingToken + BlacklistedToken
         - Delete both access_token and refresh_token cookies
         - Log LOGOUT action to AuditLog
         - Return 200 {message: "Logged out"}

       - `RefreshView` (AllowAny):
         - POST: Read refresh_token from cookie (not request body)
         - Validate token, generate new access token
         - If ROTATE_REFRESH_TOKENS: also set new refresh_token cookie
         - Set new access_token cookie with same flags
         - Return 200 {message: "Token refreshed"}
         - On invalid/expired token: return 401, delete cookies

       - `MeView` (IsAuthenticated):
         - GET: Return UserProfileSerializer(request.user)
         - Includes role, is_force_password_change

       - `PasswordResetRequestView` (AllowAny):
         - POST: Accept email. Look up user by email.
         - If found: Create PasswordResetToken (expires_at = now + 1 hour), queue Celery task to send email
         - If NOT found: return 200 anyway (don't reveal whether email exists) — security best practice
         - Return 200 {message: "If an account with that email exists, a reset link has been sent"}

       - `PasswordResetConfirmView` (AllowAny):
         - POST: Accept token (UUID) + new_password
         - Look up PasswordResetToken. Validate: not expired, not used
         - Set user password, mark token as used, clear is_force_password_change
         - Return 200 {message: "Password reset successful"}

       - `PasswordChangeView` (IsAuthenticated):
         - POST: Accept old_password + new_password
         - Validate old_password against user, validate new_password against validators
         - Set user password, clear is_force_password_change flag
         - Return 200 {message: "Password changed"}

    4. Update `backend/core/urls.py`:
       - Add URL patterns under `auth/` prefix:

         - `auth/logout/` -> LogoutView

<task type="auto">
  <name>Task 2: Rate limiting middleware — 10 failed logins per IP per 15 minutes</name>
  <files>
    backend/core/middleware.py
    backend/config/settings.py

  <action>
    1. Add to `backend/core/middleware.py` (append to existing ImpersonationMiddleware file):
       - `LoginRateLimitMiddleware`:
         - Only activates on POST to `/api/auth/login/`
         - Uses Redis (already available via CELERY_BROKER_URL / REDIS_URL env var) to track failed attempts
         - Key pattern: `login_attempts:{ip_address}` with TTL of 900 seconds (15 min)
         - On each request to login endpoint:
           - Get client IP (use get_client_ip utility from core.utils)
           - Check Redis counter for that IP
           - If counter >= 10: return 429 JSON response {"error": "Too many login attempts. Please try again in 15 minutes."}
           - Otherwise: let request through
         - After response: if response status is 401 (failed login), increment Redis counter
         - On successful login (200): delete the Redis key (reset counter)
         - Use `django.core.cache` with Redis backend OR direct redis-py connection (prefer django cache framework for consistency)

    2. Update `backend/config/settings.py`:
       - Add Django CACHES config using Redis:
         ```python
         CACHES = {
             "default": {
                 "BACKEND": "django.core.cache.backends.redis.RedisCache",
                 "LOCATION": os.environ.get("REDIS_URL", "redis://redis:6379/0"),
             }
         }
         ```
       - Add `"core.middleware.LoginRateLimitMiddleware"` to MIDDLEWARE list — AFTER ImpersonationMiddleware

    Rate limit config constants at top of middleware file:
    - `LOGIN_RATE_LIMIT_MAX = 10`
    - `LOGIN_RATE_LIMIT_WINDOW = 900`  # 15 minutes in seconds
