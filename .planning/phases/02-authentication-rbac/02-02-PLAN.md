<task type="auto">
  <name>Task 1: DRF permission classes for role-based access</name>
  <files>backend/core/permissions.py</files>
  <action>
    Create `backend/core/permissions.py` with DRF permission classes:

    1. `HasRole` — Base class that checks `request.user.profile.role.name`:
       - Returns False if user has no profile or no role
       - Subclasses define `allowed_roles` list

    2. Concrete permission classes:
       - `IsAdmin` — allowed_roles = ["Admin"]
       - `IsOperations` — allowed_roles = ["Operations"]
       - `IsFinance` — allowed_roles = ["Finance"]
       - `IsAdminOrOperations` — allowed_roles = ["Admin", "Operations"]
       - `IsAdminOrFinance` — allowed_roles = ["Admin", "Finance"]
       - `IsAnyRole` — allowed_roles = ["Admin", "Operations", "Finance"] (any authenticated user with a role)

    3. Permission matrix (document in docstring at top of file):
       - Dashboard: IsAnyRole (all 3 roles)
       - Customers: IsAdminOrOperations (Admin + Operations)
       - Jobs: IsAdminOrOperations (Admin + Operations)
       - Approvals: IsAdmin (Admin only — per freight ops separation, Admin approves)
       - Accounts: IsAdminOrFinance (Admin + Finance)
       - Reports: IsAnyRole (all 3 roles)
       - Admin panel: IsAdmin (Admin only)
       - User management: IsAdmin (Admin only)

    All permission checks use server-side UserProfile.role lookup (not JWT claims) per requirement RBAC-02.
  </action>
  <verify>
    - `grep -c "class Is" backend/core/permissions.py` shows 6+ permission classes
    - `grep "HasRole" backend/core/permissions.py` confirms base class exists
    - `grep "allowed_roles" backend/core/permissions.py` confirms role checking pattern
  </verify>
  <done>Six DRF permission classes exist, each checking UserProfile.role.name against allowed_roles. Permission matrix documented in file docstring.</done>
</task>

<task type="auto">
  <name>Task 2: Admin impersonation middleware</name>
  <files>
    backend/core/middleware.py
    backend/config/settings.py
  </files>
  <action>
    1. Create `backend/core/middleware.py`:
       - `ImpersonationMiddleware`:
         - Checks for `X-Impersonate-User` header on each request
         - If present and requesting user is Admin (check profile.role.name == "Admin"):
           - Look up target user by username (from header value)
           - Set `request._impersonated_user = target_user`
           - Set `request.user = target_user` (so DRF permission checks use impersonated user's role)
           - Store original user in `request._original_user` for audit logging
           - Log impersonation event to AuditLog: action="IMPERSONATE", user=original_user, object_repr=target_username
         - If header present but requesting user is NOT Admin: ignore header silently (don't reveal impersonation capability)
         - If target user doesn't exist: return 404 JSON response

    2. Update `backend/config/settings.py`:
       - Add `"core.middleware.ImpersonationMiddleware"` to MIDDLEWARE list — AFTER AuthenticationMiddleware (it needs request.user to exist)

    Note: The middleware only activates when the header is present. Normal requests are unaffected.
  </action>
  <verify>
    - `grep "ImpersonationMiddleware" backend/core/middleware.py` confirms class exists
    - `grep "X-Impersonate-User" backend/core/middleware.py` confirms header check
    - `grep "ImpersonationMiddleware" backend/config/settings.py` confirms middleware registered
    - `grep "_original_user" backend/core/middleware.py` confirms audit trail support
  </verify>
  <done>ImpersonationMiddleware is registered, only activates for Admin users with X-Impersonate-User header, logs impersonation events to AuditLog, and swaps request.user for permission checks.</done>
</task>
